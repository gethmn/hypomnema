## 1. Deferred Decisions — Resolutions

### Decision 1 — Pinned `sqlite-vec` crate version

**Resolved**: `sqlite-vec = "=0.1.10-alpha.3"` (the candidate stands).

**Evidence** (verified 2026-05-03 against the actual published crate tarball at `https://crates.io/api/v1/crates/sqlite-vec/0.1.10-alpha.3/download`):

- `src/lib.rs` declares `#[link(name = "sqlite_vec0")] extern "C" { pub fn sqlite3_vec_init(); }` — the exact symbol the spec assumes.
- `build.rs` is a single-line `cc::Build::new().file("sqlite-vec.c").define("SQLITE_CORE", None).compile("sqlite_vec0")` — pure cc, no system-pkg-config dependency, no external linkers required beyond a working C compiler.
- `Cargo.toml` declares `[dev-dependencies] rusqlite = "0.31.0"`, the same major.minor we already pin in our `Cargo.toml` (`rusqlite = "0.31"`).
- The crate's own integration test (in `src/lib.rs`) demonstrates the registration shape we need verbatim:

  ```rust
  use rusqlite::{ffi::sqlite3_auto_extension, Connection};
  unsafe {
      sqlite3_auto_extension(Some(std::mem::transmute(sqlite3_vec_init as *const ())));
  }
  let conn = Connection::open_in_memory().unwrap();
  let result: String = conn.query_row("select vec_version()", [], |x| x.get(0)).unwrap();
  ```

  This is the exact pattern Hypomnema will use, against the exact rusqlite version Hypomnema uses. No adjustment required.

- The crate is the package's `newest_version` and `max_version` on crates.io; published 2026-04-01 by the upstream maintainer (`asg017`); not yanked.

**Files affected**: `Cargo.toml` (add `sqlite-vec = "=0.1.10-alpha.3"`; drop `"load_extension"` from `rusqlite` features), `Cargo.lock` (regenerated by `cargo build`).

### Decision 2 — Deprecation window vs. immediate removal

**Resolved**: **Immediate removal** of `embedding.extension_path` from the config schema and `HYPOMNEMA_VEC_EXT_PATH` from the runtime contract.

**Reasoning**:

- Hypomnema is at `version = "0.7.0"` in `Cargo.toml`, has no published release on crates.io (no release process exists yet — the release-process proposal is in `notes/proposals/intake-release-process-and-changelog.md`, currently un-intaken), no `gethmn.io` install scripts, no announced public users. The "Round 6 carry-over" backlog framing also describes this as a recurring footgun for *the project's own developers*, not external operators.
- `EmbeddingConfig` already carries `#[serde(deny_unknown_fields)]` (`src/config.rs:159`). Dropping `extension_path` from the struct produces a serde error of the form `unknown field 'extension_path', expected one of 'endpoint', 'model', 'dimension', 'api_key', 'timeout_ms', 'max_retries', 'batch_size'`. That message is already actionable for an existing operator (it names the field by name); supplementing it with custom diagnostic text is overengineering for a pre-public-release schema.
- The spec's preferred answer is immediate removal "before public release"; the project clearly meets that bar.
- Story 3 has explicit acceptance criteria for the immediate-removal branch ("a config containing `embedding.extension_path` fails with an actionable unknown-field message that says sqlite-vec is bundled"). The serde error names the field; the docs and manual-testing fixture call out the migration path.

**Files affected**: `src/config.rs` (drop the field, default fn, `VEC_EXT_PATH_ENV` const, `resolved_extension_path()` method); `tests/config.rs` (add a regression test for the unknown-field rejection); `docs/reference/configuration.md` (drop both the example and the env-var mapping table row); `notes/manual-testing/00-setup.md` (drop the sqlite-vec section); `notes/backlog.md` (strikethrough closure).

### Decision 3 — Compile-time feature flag for dynamic loading

**Resolved**: **No flag.** Static linkage is the only path.

**Reasoning**: No concrete maintainer workflow surfaced during research. The crate's build is fast (one C source, ~9k LOC), the dev shell already has the C toolchain (it has `pkg-config`, and on Darwin the Apple SDK is on stdenv — see `flake.nix`), and CI's compiled artifact is the production artifact. A flag would expand the test matrix (or worse, leave a flag-gated path untested) for a benefit nobody has asked for. AGENTS.md "Don't add features beyond what the task requires" applies directly.

**Files affected**: none (default to absence). If a maintainer ever needs a dynamic-loading escape hatch, that is a future backlog item, not a workplan-time concession.

### Q2 — Are there existing external users on the `extension_path` contract?

### Q3 — Does the macOS CI runner have a working C toolchain for sqlite-vec's build script?

**Answer**: Yes. `macos-latest` runners ship Xcode Command Line Tools by default — clang is available out of the box. `ubuntu-latest` ships `gcc`/`build-essential` on the standard image. No new CI setup step is required; the existing "Install sqlite-vec extension" step is removed wholesale (see Task 4.1). If a future runner-image regression breaks this, `cc` will fail at build time and CI will fail loudly — no silent fallback.

### Q4 — Does `flake.nix` provisioning of sqlite-vec serve any purpose beyond the loadable extension?

**Answer**: There is no `flake.nix` sqlite-vec dylib provisioning to remove. Verified by reading `flake.nix` end-to-end: the `devShells.default` package list is `rustToolchain, cargo-watch, cargo-nextest, cargo-edit, bacon, just, sqlite, python312, uv, pkg-config, git-cliff` plus `darwinDeps = [libiconv]`. No `sqlite-vec` package, no dylib path, no setup hook that downloads sqlite-vec. The `notes/backlog.md` § Operational follow-ups entry "`flake.nix` sqlite-vec dylib provisioning" describes a *gap* — the dev shell never handled the sqlite-vec dylib, and any developer running tests outside CI had to provision it manually. After this round, the gap closes by becoming irrelevant: no dylib is needed at all. Action: strikethrough the backlog entry; no `flake.nix` code change.

The `pkg-config` and `sqlite` packages stay in `flake.nix`. `pkg-config` is used by other Rust crates in our graph; the `sqlite` package gives developers the `sqlite3` CLI for ad-hoc store inspection — neither is sqlite-vec-specific.

### Phase 2 — Static registration + store rewire (3 tasks; Stories 1, 2)

#### Task 2.1 — Add a process-once `register_sqlite_vec()` initializer

- **Files**: `src/lib.rs` (new public function), or a new `src/store/sqlite_vec_init.rs` module exposed through `src/store/mod.rs` and re-exported on `src/lib.rs`.
- **Recommended shape** (single internal module, public re-export from `lib.rs`):

  ```rust
  // src/store/sqlite_vec_init.rs
  use std::sync::Once;
  static REGISTER: Once = Once::new();

  /// Register sqlite-vec's static init function with SQLite for every
  /// connection opened from this process. Idempotent; safe to call from
  /// multiple call sites. Must run before the first rusqlite Connection is
  /// opened.
  pub fn register_sqlite_vec() {
      REGISTER.call_once(|| {
          // Per the upstream sqlite-vec 0.1.10-alpha.3 integration test:
          // signature transmute is required because rusqlite's binding
          // expects `Option<unsafe extern "C" fn(...) -> c_int>` while
          // sqlite-vec's symbol is plain `extern "C" fn()`.
          unsafe {
              rusqlite::ffi::sqlite3_auto_extension(Some(std::mem::transmute(
                  sqlite_vec::sqlite3_vec_init as *const (),
              )));
          }
      });
  }
  ```

- **Re-export**: `pub use store::register_sqlite_vec;` in `src/lib.rs`, so integration tests under `tests/` can call `hypomnema::register_sqlite_vec()` directly.
- **Acceptance**:
  - Function compiles, is `pub`, and its module is wired through `src/lib.rs`.
  - `cargo clippy --all-targets -- -D warnings` is clean for the new module (the `unsafe` + `transmute` block needs a `#[allow(clippy::missing_transmute_annotations)]` or an explicit cast — match the upstream test's shape; a one-line `// SAFETY:` comment naming the registration contract is the only comment to add).
- **Risk**: medium (this is the spec's "main new correctness boundary").
- **Depends on**: Task 1.1.

#### Task 2.3 — Add a two-independent-connection probe test for `vec_version()`

- **Files**: new test in `src/store/mod.rs` (in-module `#[cfg(test)] mod tests`), naming it `vec_version_succeeds_on_two_independent_connections`.
- **Shape** (illustrative):

  ```rust
  #[tokio::test]
  async fn vec_version_succeeds_on_two_independent_connections() {
      let dir = tempdir().unwrap();
      let vault_id = VaultId::new();
      let store = Store::open(&vault_id, dir.path(), "index.sqlite", &EmbeddingConfig::default())
          .await
          .unwrap();
      let pool1 = store.pool();
      let pool2 = store.pool();
      // Force two distinct connections from the pool. The pool max is 8;
      // holding `c1` while requesting `c2` proves a second connection is
      // opened, not the same one reused.
      let v1: String = run_blocking(move || {
          let c1 = pool1.get().unwrap();
          let c2 = pool1.get().unwrap();
          let v_a: String = c1.query_row("SELECT vec_version()", [], |r| r.get(0)).unwrap();
          let v_b: String = c2.query_row("SELECT vec_version()", [], |r| r.get(0)).unwrap();
          assert_eq!(v_a, v_b, "vec_version must agree across pool connections");
          v_a
      }).await;
      assert!(v1.starts_with('v'), "vec_version() returned {v1:?}");
      // Reuse `pool2` to confirm registration survives the first pool's drop pattern.
      let v2: String = run_blocking(move || {
          let c = pool2.get().unwrap();
          c.query_row("SELECT vec_version()", [], |r| r.get(0)).unwrap()
      }).await;
      assert_eq!(v1, v2);
  }
  ```

- **Acceptance**:
  - Test passes with the static registration in Task 2.1 in place.
  - Test fails (in a controlled probe by a builder, not as a permanent state) if `register_sqlite_vec()` is omitted or moved after `pool::build_pool` — confirms the registration-order boundary is real.
- **Risk**: low (test-only).
- **Depends on**: Task 2.2.

## 6. Sequencing Rationale

The phase order is dictated by the build graph:

1. **Phase 0 baseline** must run on the *unchanged* tree to record pre-round size.
2. **Phase 1 (Cargo)** must precede Phase 2: `register_sqlite_vec()` cannot reference `sqlite_vec::sqlite3_vec_init` until the crate is on the dependency graph.
3. **Phase 2 (registration + pool)** must precede Phase 3 (config): Task 3.1 deletes `resolved_extension_path()`, which is a public method that `src/store/mod.rs` currently calls. Doing Phase 3 first would break the build.
4. **Phase 3 (config + tests)** must precede Phase 4 (CI): until tests under `tests/` no longer require `EmbeddingConfig::default().resolved_extension_path()` and no longer load a dylib from `~/.local/share/hypomnema`, dropping the CI install step would red the test job.
5. **Phase 4 (CI)** must precede Phase 5 (LDS canon) in spirit: until CI passes green on the static path, the docs claiming "CI no longer downloads tarballs" would be a lie. In practice the two phases can run in parallel branches but must merge together; the round bundles them per intake § "Proposed Roadmap Shape".
6. **Phase 6 (gate)** runs last by definition.

Within Phase 2 the ordering is also tight: Task 2.3 (the two-connection probe) must come *after* Task 2.2 (registration call wired) but is the primary safety net for catching a registration-order regression in Phase 3 and beyond. Treat 2.3 as a permanent fixture in the test suite, not just a workplan-time check.

**Risk-mitigation gates**:

- After Task 2.3, builders run the full `cargo nextest run` before touching config. If 2.3 passes but other store tests fail with `vec_version() is unavailable`, that is a real registration-order miss in some other call site and is the round's highest-priority concern.
- After Task 3.2, builders run `tests/embedding.rs` and `tests/mcp.rs` specifically — these are the broadest end-to-end coverage.
- Task 6.1's release-build + size check serves as the final integration smoke test.

**Independence**: Tasks within Phase 5 are file-disjoint and can be parallelized across builders; sequencing only matters within phases 1–4.
